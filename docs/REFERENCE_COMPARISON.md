# Reference Comparison

This note compares LakatoTree with the external systems used as engineering
references. The decision is deliberate: borrow vocabularies and export formats,
but keep the Lakatos verdict gates inside this engine.

## Summary

| Reference | Strong point | Missing for LakatoTree | Engine decision |
|---|---|---|---|
| OpenLineage | Standard event vocabulary for `Run`, `Job`, and `Dataset` lineage | No Lakatos verdict, critique, or raw replay gate | Export optional RunEvents from `Derivation` and `LineageReplayResult` |
| Marquez | OpenLineage-compatible backend and lineage graph UI | Backend dependency; not a theory/verdict engine | Treat as optional sink for OpenLineage events |
| DVC pipelines | Raw deps, commands, params, outputs, lock hashes, replay model | Not a critique/Lakatos programme model | Export DVC-style stage and lock manifests |
| W3C PROV / Python `prov` | Generic `Entity`, `Activity`, `Agent` provenance semantics and serializations | Too generic; no domain gates by itself | Export PROV-like documents from observations, bash acts, and derivations |
| MLflow Dataset Tracking | Experiment params, metrics, artifacts, dataset lineage | Metrics can be logged without proving progressive shift | Use as auxiliary experiment mirror only |
| NetworkX / Neo4j | In-memory graph algorithms / persistent queryable graph | Neither defines Lakatos gate semantics | Keep pure tests DB-free; mirror long-term KG into Neo4j |

## OpenLineage / Marquez

OpenLineage defines a shared way to collect lineage metadata about jobs, runs,
and datasets. Marquez is the reference-style backend/UI path for storing and
visualizing those events.

LakatoTree mapping:

```text
Derivation/PipelineRun -> OpenLineage RunEvent
producer script        -> Job
inputs/output artifacts -> Dataset
LineageReplayResult    -> ordered RunEvent sequence
```

Implemented:

- `lakatos.adapters.derivation_to_openlineage_event`
- `lakatos.adapters.lineage_result_to_openlineage_events`

Why not make OpenLineage canonical? Because a lineage event can say "this job
produced that dataset" without deciding whether the branch is progressive,
degenerating, blocked by critique, or replayable from raw roots.

Sources:

- https://openlineage.io/
- https://openlineage.io/getting-started/
- https://marquezproject.ai/

## DVC

DVC is the closest practical reference for the BPC concern: raw data changes,
buffers exist, and final outputs must be reproducible from recorded stages.

LakatoTree mapping:

```text
RawDataArtifact -> DVC dependency
Derivation      -> dvc.yaml stage
output_sha      -> dvc.lock output hash
RebuildRecipe   -> dvc repro-like replay plan
```

Implemented:

- `lakatos.adapters.derivations_to_dvc_pipeline`
- `lakatos.adapters.derivations_to_dvc_lock`
- `lakatos.adapters.rebuild_recipe_manifest`
- `lakatos.lineage.DatasetManifest`
- `lakatos.lineage.verify_dataset_manifest`

Why not require DVC? BPC pipelines can span Windows capture machines, HALCON,
Zivid/ZDF roots, generated caches, and solver packages outside a clean DVC
project. The engine should export DVC-compatible structure without requiring DVC
to be installed.

Sources:

- https://doc.dvc.org/start
- https://doc.dvc.org/start/data-pipelines/data-pipelines

## W3C PROV / Python `prov`

PROV gives the semantic floor: data and things are entities, processes are
activities, and responsible actors/tools are agents.

LakatoTree mapping:

```text
InternetObservation -> Entity generated by WebFetch Activity
BashAct             -> Activity generating BashResult Entity
Derivation          -> Activity using input Entities and generating output Entity
producer script     -> Agent
```

Implemented:

- `lakatos.adapters.derivations_to_prov_document`
- `lakatos.adapters.observation_to_prov_document`
- `lakatos.adapters.bash_act_to_prov_document`

Why not depend on Python `prov` now? Plain dicts keep the core engine testable
without optional graph/serialization dependencies. A serializer adapter can
convert these dicts to `prov` documents later.

Sources:

- https://www.w3.org/TR/prov-dm/
- https://pypi.org/project/prov/

## MLflow Dataset Tracking

MLflow is useful for experiments: params, metrics, artifact paths, datasets, and
runs. It is not enough for LakatoTree verdict authority.

LakatoTree should use MLflow only as an optional telemetry mirror:

```text
PipelineRun.params -> MLflow params
Lakatos metric     -> MLflow metric
Derived artifact   -> MLflow artifact
DatasetManifest    -> MLflow dataset input
```

The authoritative verdict remains:

```text
LakatosGate + CredibilityPromotionGate + LineageReplayGate + critique standing
```

Source:

- https://mlflow.org/docs/latest/ml/dataset/

## NetworkX / Neo4j

LakatoTree already has pure in-memory tree/graph logic. That should stay:

- Unit tests should not require Neo4j.
- Graph algorithms can use NetworkX-style data structures when needed.
- Neo4j is the long-lived KG mirror for queries, audits, and cross-project
  history.

## Upgrade Result

The adapter layer now gives the engine a clean bridge to the outside world:

```text
LakatoTree pure gates
  -> OpenLineage/Marquez for lineage ecosystem integration
  -> DVC-style manifests for raw-root replay
  -> DatasetManifest + EnvironmentFingerprint for G-RebuildFromRaw checks
  -> PROV-like documents for standard provenance semantics
  -> MLflow-compatible telemetry later, without verdict authority
```

This keeps the hard core intact: external systems are evidence and transport,
not judges.
